# 1. 🏆 하프 AI 동반자 시스템 (PRD 졸업)

## 졸업 위치
- **PRD v5 Part 11.5**: 메시지의 발신자 — 하프 (HARP)
- **PRD v5 Part 11.6**: 하프 메시지 풀 카테고리 체계
- **PRD v5 Part 11.7**: 하프의 톤 가이드라인
- **PRD v5 Part 11.8**: 메시지 풀 우선순위 시스템
- **PRD v5 Part 19**: `harp_messages`, `user_message_history` 테이블
- **BUSINESS_MODEL v2 5.6**: 하프 시스템 Phase별 로드맵

## 발견 시점
2026-04-11, Sprint 1 시작 직전

## 핵심 아이디어

소설 속 **하프(AI)**가 주인공 **덕훌**에게 퀘스트를 주고, 행동을 보고, 경험치를 주고, 레벨업시키는 그 시스템 — **그것을 앱에 그대로 구현**한다.

독자가 소설을 읽으며 "나도 하프 같은 AI 동반자가 있으면 좋겠다"고 느끼는 순간, 앱을 열면 **진짜로 하프가 거기 있다**.

## 왜 결정적인가

이건 단순한 기능 추가가 아니다. **LV UP의 모든 차별점을 한 점으로 모으는 메커니즘**이다.

| 다른 자기계발 앱 | 하프 AI가 있는 LV UP |
|---|---|
| 차가운 UI 텍스트 | 살아있는 캐릭터의 말 |
| "퀘스트 완료" 알림 | 하프의 인정 |
| 사용자가 시스템과 대화 | 사용자가 하프와 대화 |
| 텍스트만 있는 IP | IP가 앱 안에서 실제로 살아있음 |
| 마케팅과 제품이 분리 | 소설을 읽는 것 = 제품 체험 |

### 가장 큰 무기

**소설의 주인공이 받는 경험을 독자가 그대로 받을 수 있다.**

- LiFE RPG도 못 함
- Habitica도 못 함
- 그 어떤 자기계발 앱도 못 함
- LV UP만 가능 — 왜냐하면 **소설이 있기 때문에**

## 양방향 유입 구조

```
소설 독자 → "하프 같은 AI가 갖고 싶다" → 앱 가입 → 하프를 만남 → 더 깊은 몰입
        ↑                                                            ↓
        └──────  앱 사용자 → "하프가 누구야?" → 소설 검색 → 독자됨  ←
```

두 IP가 서로를 강화한다. 기존 IP-앱 관계는 일방향(IP → 앱)이지만, LV UP은 양방향이다.

## 구현 전략 (비용 함정 회피)

⚠️ **함정**: AI를 모든 인증에 쓰면 비용이 폭발한다.

### 90/10 원칙

**90%는 사람이 쓴 메시지 풀**:
- 작가(창업자)가 직접 작성
- 카테고리별 수십~수백 개 변형
- 가입 환영, 첫 퀘스트, 보상, 레벨업, 복귀, 격려 등
- LLM 호출 0회
- 품질이 더 좋음 (작가의 손길 = LV UP 세계관 정확도)

**10%만 실제 LLM 호출**:
- 사용자가 하프에게 직접 질문할 때 (제한 있음)
- 특별한 순간 (10레벨, 시즌 끝, 1주년 등)
- 주간 회고 시 행동 요약
- **유료 사용자만 가능**

### 비용 시나리오 비교

| 접근 | 사용자 1만 명 시 월 비용 | 매출 잠식도 |
|---|---|---|
| 모든 인증을 AI verdict | ₩900~2,700만 | 💀 망함 |
| 풀 대본 + 유료 AI 대화 | ₩50~150만 | 🟡 가능 |
| 퀘스트 제안만 AI (유료) | ₩10~30만 | ✅ 안전 |

## 메시지 풀 카테고리 초안

작가가 작성해야 할 메시지 카테고리:

```
HARP_MESSAGES = {
  // 입장
  welcome: {
    first_time: [50개 변형],
    returning_after_break: [30개],
    streak_continuation: [20개],
  },
  
  // 퀘스트
  quest_suggestion: {
    morning: [30개],
    after_failure: [20개],
    streak_milestone: [15개],
  },
  
  // 보상 (스탯별)
  reward: {
    vitality: [30개],
    focus: [30개],
    execution: [30개],
    knowledge: [30개],
    relationship: [30개],
    influence: [30개],
  },
  
  // 특별 순간
  special: {
    level_up: [20개],
    title_change: [15개],
    seven_day_streak: [10개],
    thirty_day_streak: [10개],
    first_failure_compassion: [15개],
  },
  
  // 회고
  weekly_review: {
    great_week: [15개],
    okay_week: [15개],
    tough_week: [15개],
    first_week: [10개],
  },
}
```

총 약 500~700개 메시지. **이게 LV UP IP의 일부**가 된다. 작가가 직접 쓰는 게 가치다.

## 핵심 원칙

1. **사용자는 LLM 호출과 풀 메시지를 구분 못 한다** — 둘 다 "하프가 한 말"
2. **하프는 평가하지 않는다, 함께한다** — 소설에서 하프의 본질
3. **모든 메시지는 작가의 톤과 일치해야 한다** — IP 일관성
4. **AI 비용이 매출의 30%를 넘으면 안 된다** — 단위경제 원칙

## 구현 시점

**Sprint 1~5 (Phase 0)**: 구현 안 함. 단순 텍스트만.

**Phase 1 후반 (M5~6)**: 메시지 풀 시스템 도입
- 작가가 메시지 100~200개 작성
- 보상 메시지를 하프 메시지 풀로 교체
- 캐릭터 이미지/이름은 아직 안 나옴 (그냥 "다음 행동" 형태)

**Phase 2 (M6~12)**: 하프 캐릭터 등장
- 하프 캐릭터 일러스트
- 하프의 이름과 인격이 명시적으로 드러남
- 메시지 풀 500개+
- 소설과 명시적 연결 ("하프가 덕훌에게 했던 것처럼")

**Phase 3 (M12~18)**: 유료 AI 대화
- Founder's Circle 멤버는 하프와 직접 대화 가능
- 일일 제한 (3~5회)
- 대화 내용은 하프의 톤 가이드라인 시스템 프롬프트 사용

## 검증 질문

이 시스템이 작동하는지 확인할 질문들:

- [ ] 사용자가 "하프"라는 이름을 자연스럽게 인식하는가?
- [ ] 메시지가 LV UP 세계관과 일치한다고 느끼는가?
- [ ] 하프 때문에 소설을 찾아보는 사용자가 있는가? (양방향 유입)
- [ ] 풀 메시지와 LLM 메시지의 만족도 차이가 있는가?
- [ ] AI 비용이 통제 가능한 수준인가?

## 위험 요소

1. **메시지 풀이 반복적으로 느껴짐**
   - 해결: 충분히 많은 변형 + 컨텍스트 기반 선택
2. **사용자가 LLM을 기대하는데 풀 메시지여서 실망**
   - 해결: 처음부터 "하프는 시스템입니다"로 명시 (AI 사칭 X)
3. **소설 안 본 사용자에게 의미 없음**
   - 해결: 앱에서도 하프 캐릭터를 자연스럽게 소개
4. **작가 한 명이 메시지 500개 작성하는 부담**
   - 해결: 한 번에 안 쓰고 분기마다 50~100개씩 추가

# 2. AI 퀘스트 제안 (하프의 첫 기능)

## 핵심 아이디어

사용자가 하프에게 자연어로 말하면, 하프가 적절한 퀘스트를 자동으로 만들어준다.

```
사용자: "오늘 1시간 운동했어"
하프: "달리기였나요? 그렇다면 [오늘의 달리기 30분] 일일 퀘스트를 만들어드릴게요. 
       체력 +3, 지구력 +2 보상이 어울려 보입니다."
사용자: "응 좋아"
하프: [퀘스트 자동 생성 + 자동 인증 + 보상 지급]
```

## 왜 강력한가

LV UP의 가장 큰 마찰점은 **사용자가 직접 퀘스트를 만들어야 한다**는 거다. 이걸 AI가 해주면:

- 진입 장벽 사라짐
- 자연스러운 대화로 모든 게 됨
- 하프와의 첫 상호작용 = 즉각적 가치

## 비용 효율성

**Verdict와 다르다**:
- Verdict: 모든 인증마다 호출 (사용자 1만 → 월 90,000회)
- 퀘스트 제안: 사용자가 명시적으로 요청할 때만 (사용자 1만 → 월 5,000~10,000회)
- **약 10배 적은 호출**

## 구현 시점

Phase 2 후반 또는 Phase 3 초반.
Founder's Circle 전용 기능으로 시작.

# 3. 유료 AI Verdict (조건부 검토)

## 핵심 아이디어

커뮤니티 verdict가 늦거나 안 될 때, 유료 옵션으로 AI가 즉시 verdict.

## 가능한 모델

- 무료: AI Verdict 없음
- Founder's Circle (월 9,900원): 월 10회 무료
- 추가: 1회 500원 또는 패키지

## 우려 사항

⚠️ **본질적으로 비싼 기능이다**. 사용자가 늘면 비용이 비례 증가.

### 영리한 변형: 하이브리드 Verdict

- AI는 즉시 1차 verdict ("이 인증은 유효해 보입니다, +15 XP")
- 정확한 verdict는 여전히 커뮤니티 (나중에 보너스 XP)
- AI가 틀려도 큰 문제 없음 (커뮤니티가 보정)
- AI는 "기본 보상", 커뮤니티는 "보너스 검증"

이렇게 하면 AI 사용을 최소화하면서도 즉시 보상의 만족감을 줄 수 있다.

## 구현 시점

**Phase 3 이후**. 그 전에는 검토만.
사용자가 "verdict 늦어서 답답하다"는 피드백이 충분히 모일 때만 도입.

# 4. 🏆 소설-앱 양방향 유입 구조 (PRD 졸업)

## 졸업 위치
- **PRD v5 Part 16.2 화면 5a**: 첫 퀘스트 보상 연출 특별 케이스 — 덕훌 1화 읽기 링크
- **PRD v5 Part 15.1**: 첫 사용 흐름 12번 — 보상 화면 하단 1화 링크
- **PRD v5 character_courses 테이블**: 소설 외전 해금 시스템
- **BUSINESS_MODEL v2 3.2**: Founder's Circle 혜택 2번 — 덕훌 성장 코스

## 핵심 아이디어

LV UP의 IP와 앱은 단순히 마케팅 관계가 아니라, **서로를 강화하는 양방향 관계**가 되어야 한다.

## 현재 일반적인 IP-앱 관계 (일방향)

```
IP → 팬덤 → 앱 사용자
```

문제: IP를 모르는 사람은 앱에 안 옴.

## LV UP의 양방향 관계

```
소설 독자 ──→ "하프가 갖고 싶다" ──→ 앱 가입 ──→ 하프 만남
                                                       ↓
앱 사용자 ←── "하프가 누구야?" ←── 캐릭터 등장 ←─────┘
        ↓
    소설 찾아봄
        ↓
    소설 독자
```

## 구현 아이디어

### 소설 → 앱
- 소설 각 화 끝에 "현실 퀘스트" 코너
- "덕훌이 받은 [집중력 훈련] 퀘스트, 너도 받을래? → lvup.world"
- QR 코드 또는 직접 링크

### 앱 → 소설
- 앱 안에 "하프 이야기" 섹션
- "하프는 누구인가? 덕훌의 성장 이야기 1화 읽기"
- 소설 1~3화는 앱에서 무료
- 4화부터는 소설 플랫폼으로 (Patreon, 본인 사이트, 추후 카카오페이지)

### 시즌제 통합
- 소설 시즌 ↔ 앱 시즌 동기화
- 소설에서 덕훌이 새로운 챕터를 시작하면, 앱에도 새 시즌 시작
- 독자와 사용자가 같은 흐름으로 성장

## 검증 질문

- 앱 사용자 중 소설을 읽기 시작한 비율은?
- 소설 독자 중 앱에 가입한 비율은?
- 양방향 전환율의 차이는?

## 구현 시점

**Phase 2 (M6~12)**: 기본 양방향 링크
**Phase 3 (M12~18)**: 시즌 동기화

# 4.5 🏆 덕훌 성장 코스 (신규 PRD 졸업)

## 졸업 위치
- **PRD v5 Part 19**: `character_courses`, `character_course_quests`, `user_course_progress` 테이블
- **PRD v5 Part 8.1**: 퀘스트 종류에 `character_course` 추가
- **BUSINESS_MODEL v2 3.2**: Founder's Circle 혜택 2번

## 핵심 아이디어

소설 주인공 덕훌이 LV.0 → LV.5까지 통과한 퀘스트를 **사용자가 1:1로 
자기 인생에 복사해서 실행할 수 있는** 시스템. 퀘스트 완료 시 소설 외전 
단락이 보상으로 해금됨.

## 왜 결정적인가

일반 자기계발 앱의 캐릭터 차용은 "장식"이지만, 덕훌 성장 코스는 
**소설 속 픽션을 사용자가 실제 행동으로 복사하는 전례 없는 형태**다.

- 독자가 "나도 덕훌처럼 해보고 싶다"고 느낀 순간의 가장 직접적인 답
- 퀘스트 완료 = 소설 외전 해금 = 소설을 더 깊이 읽는 유일한 방법
- 작가에게도 이득: 본편에 넣기 어려운 세부 디테일을 외전으로 빠져나갈 자리 확보

## 확장 가능성

첫 코스는 덕훌이지만, 구조적으로 **다른 캐릭터 코스도 확장 가능**:
- **허민 코스**: 혼자 깊이 파고드는 법 (탐구자 대상)
- **마지석 코스**: 시스템을 움직이는 법 (주도자 대상)
- 5클래스 진단과 연결되어 사용자에게 맞는 코스 추천

## 접근 권한

Founder's Circle 전용 (결정 4, 11). 무료 사용자는 접근 불가.
이것이 결제 동기의 핵심 중 하나 — "소설을 더 깊이 읽는 유일한 방법".

## 큐레이션 작업

덕훌의 모든 퀘스트가 일반 사용자가 따라할 수 있는 건 아니다.
(부산-서울 도보 같은 건 못 따라함)
시즌 1 집필 완료 후 "이 퀘스트는 코스에, 이건 본편 전용"으로 분류하는 
큐레이션 작업 필요.

## 구현 시점

- Phase 1: 데이터 모델만 박아둠 (마이그레이션 비용 절감)
- Phase 2: 덕훌 코스 활성화, 소설 1화~5화 외전 작성
- Phase 3: 다른 캐릭터 코스 확장 (허민, 마지석 등)

# 5. 기타 떠오른 아이디어들 (🌱 새싹)

검토가 더 필요하지만 잊지 말아야 할 아이디어들.

## 5.1 길드 시스템

PRD에 이미 있지만 더 구체화 필요.

- 같은 클래스끼리의 길드 (Builder 길드, Creator 길드 등)
- 같은 목표끼리의 길드 (운동 길드, 독서 길드)
- 길드 공동 퀘스트
- 길드원끼리만 보이는 사회적 압력

## 5.2 시즌 코호트 게이미피케이션

BUSINESS_MODEL의 시즌 코호트를 더 게임처럼.

- 시즌 시작 시 "출항" 연출
- 4주 동안 "여정" 메타포
- 졸업 = "도착"
- 졸업장이 아니라 "여정의 증표" 디지털 굿즈

## 5.3 사용자 생성 퀘스트의 큐레이션

다른 사용자가 만든 좋은 퀘스트를 발견하고 가져올 수 있는 시스템.

- "이번 주 인기 퀘스트"
- "비슷한 클래스의 사람들이 하는 퀘스트"
- 큐레이션은 AI보다 사람이 (커뮤니티 좋아요)

## 5.4 오프라인 미팅

LV UP 사용자들의 정기 오프라인 모임.

- 분기 1회 서울에서 시작
- "Genesis Night" — 창업자와 만나는 자리
- 시즌 졸업식과 결합
- Phase 2 후반에 검토

## 5.5 LV UP 다이어리

종이 다이어리 굿즈. 앱과 연동.

- 매일 한 줄 기록 + 사진 한 장
- 디지털과 아날로그의 결합
- 굿즈 매출 + 브랜드 강화
- Phase 3 이후

## 5.6 AI 코칭 (1:1 한정)

가장 비싼 AI 기능. 가장 마지막에.

- 월 5명 한정, 회당 30~50만원
- 사용자의 1주일 데이터를 AI가 분석
- 개인화된 다음 주 계획 제안
- 창업자 1:1 코칭과 함께 운영
- Phase 3 후반

## 5.7 LV UP API

다른 개발자들이 LV UP과 통합할 수 있는 API.

- 운동 앱 → LV UP 자동 인증
- 독서 앱 → LV UP 자동 인증
- 깃허브 커밋 → LV UP 자동 인증
- 개발자 생태계 형성 가능
- Phase 4 이후

## 5.8 학교/기업 단체 프로그램

LV UP for Schools, LV UP for Teams 확장.

- 교사가 학생들에게 퀘스트 제공
- 팀장이 팀원들에게 챌린지 제공
- B2B 매출 + 사용자 확보 채널
- Phase 3 이후

# 7. 아이디어 평가 기준

새 아이디어가 들어오면 다음 기준으로 평가한다.

## 7.1 LV UP 정체성 일치

- [ ] 5대 차별화 원칙에 부합하는가? (도구 X 세계, 숫자 X 이야기, 스펙 X 행동, 비교 X 자기성장, 멈춤도 성장)
- [ ] "절대 하지 않을 것" 목록에 안 걸리는가?
- [ ] LV UP의 톤(따뜻함, 진정성, 게임적 재미)에 맞는가?

## 7.2 단위경제

- [ ] 사용자 1만 명 시 운영 비용이 매출의 30% 이하인가?
- [ ] 한계 비용이 0에 가까운가, 아니면 사용자에 비례하는가?
- [ ] 비용을 사용자에게 직접 전가할 방법이 있는가?

## 7.3 차별성

- [ ] 다른 자기계발 앱에 없는 것인가?
- [ ] 있다면, LV UP이 더 잘할 수 있는 이유가 있는가?
- [ ] LV UP만의 IP/세계관과 결합되어 있는가?

## 7.4 구현 난이도 vs 임팩트

- [ ] 구현 난이도: 낮음 / 중간 / 높음
- [ ] 임팩트: 작음 / 중간 / 큼
- [ ] 우선순위: 임팩트 ÷ 난이도

# 8. 아이디어가 졸업하는 길

`IDEAS.md`에서 다른 문서로 옮겨가는 흐름.

```
🌱 새싹 (떠오름)
   ↓
🌿 검토 중 (Phase 결정, 비용 분석)
   ↓
🌳 검증 후보 (Phase 진입 시 실험 대상)
   ↓
🏆 PRD 졸업 (정식 기능으로 PRD/BUSINESS_PLAN에 편입)
```

또는:

```
🌱 새싹 → 🍂 보류 (현재 LV UP에 안 맞음)
```

졸업한 아이디어는 IDEAS.md에서 짧은 요약 + 졸업 일자만 남기고 본문은 PRD로 이동.

# 마지막 한 마디

이 문서의 가장 중요한 항목은 **하프 AI 동반자 시스템**이다.

이건 단순한 기능이 아니라 **LV UP을 LV UP이게 만드는 결정적 메커니즘**이다.
다른 모든 자기계발 앱이 가질 수 없는 단 하나의 것.

소설을 쓰는 작가가 만드는 자기계발 앱이라서 가능한 것.
그래서 LV UP을 만드는 사람이 작가여야 하고, 작가가 만들어야 한다.

**이 한 가지를 잊지 말 것.**
